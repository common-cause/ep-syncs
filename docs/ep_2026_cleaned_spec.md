# ep_2026_cleaned — The 2026 EP Interface Layer

*As-built 2026-07-23. SQL: `bq/ep_2026_cleaned/` (applied by
`apply_bq_views.py`; the 3x union views are generated by
`airtable_views.py`). Sources: `ptv_raw_2026` (PTV) and `ep_2026_raw`
(Airtable capture — contract in `docs/airtable_bases_sync_spec.md`).*

**This dataset is THE interface other projects query for 2026 EP
recruitment + monitoring data.** Point new work here, not at the raw
datasets.

## Dataset invariants

- Views + UDFs only — no tables, no PII copies. Always current (no
  scheduled rebuilds; freshness = the feeding syncs' cadence, see
  `sync_health`).
- Every object carries `OPTIONS(description=...)` (flows into the schema
  MCP docs).
- **Emails and phones are already normalized** (`norm_email` = TRIM+LOWER,
  blank→NULL; `norm_phone` = digits-only last-10, blank→NULL). Consumers
  never re-normalize and should join on these values. `email_raw` /
  `volunteer_email_raw` keep the as-delivered value where a consumer needs
  it (e.g. Airtable upsert keys, which match exactly).
- `state` on Airtable-derived rows comes from the base registry, never
  from record fields.
- Airtable-derived views regenerate automatically after every capture run
  (a base going live appears the same morning).

## The union mechanism (generated views)

`airtable_views.py` holds one canonical contract per entity and renders a
UNION ALL branch per registered base from live inputs (registry +
`ep_2026_raw.INFORMATION_SCHEMA.COLUMNS`, so casts are exact). Per-column
resolution order:

1. registry `canonical_overrides` (`{"entity": {"col": "source" | null}}`)
2. default source candidates (tuned to the 2026 templates)
3. heuristics (quiz grader regex; `shift`/`what_type_of_poll_shift...`)
4. record-link fallback (checklist `volunteer` link → base's Shifted
   Volunteers; `polling_places` link → base's Polling Places)
5. NULL-pad (`CAST(NULL AS T)`)

Entities with no live base render as the full contract + `WHERE FALSE`.
Committed `3x_*.sql` files are review snapshots — refresh with
`python apply_bq_views.py --render-generated` and commit (part of the
base go-live checklist); `--check` flags snapshot-vs-deployed drift.

## Verification suite (rerun after material changes)

- **V1 grain**: 0 dupes at each view's declared grain.
- **V2 reconciliation**: `volunteers WHERE is_active AND in_ptv` per state
  == `v_users_current` per state (exact match on 2026-07-23, 51 states).
- **V3 normalization**: no exposed email/phone fails the norm regexes;
  shift↔volunteers join loss ≤ known non-registrants (14 people national
  on 2026-07-23; raw-email joins would lose more — the bug this layer
  fixes).
- **V4 union coverage**: every enabled registry (base, entity) appears as
  a branch (generator logs branch counts; compare vs registry).
- **V5 drift**: `python apply_bq_views.py --check` clean.
- **V6 quiz mapping**: per quiz base, score non-null ≈ 100% and score_max
  present (catches grader-column renames; fix via `canonical_overrides`).
- **V7 consumer migration**: row-for-row comparison before flipping any
  consumer query (see below).

## Known deferred items

- **`ptv_raw_2026.v_shift_volunteers_current` normalization fix**: the raw
  view groups by raw email and passes phone through un-normalized. Fixing
  it in place would silently re-key the shift sync's Airtable upsert
  (exact-match performUpsert) and orphan mixed-case Airtable records — it
  needs a coordinated one-time Airtable dedupe first. The interface layer
  normalizes independently, so nothing here blocks on it. Track as its own
  change if raw-view consumers need it.
- Cross-state person stitching (a person_id spanning states) — v1 keys on
  (state, email); `volunteer_activity` makes cross-state lookups by email
  easy without it.
